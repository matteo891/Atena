## What

**Veto ROI R-08 — predicato scalare**: primo gate applicativo del Tetris. Inaugura `src/talos/vgp/` (vuota fino ad ora; FILE-ADR-MAP riga 77 governa "normalize, score, veto" sotto questa cartella).

R-08 verbatim (PROJECT-RAW.md riga 226):

> **R-08 — VETO ROI MINIMO.** Nonostante la monarchia del VGP, viene applicato un filtro di sbarramento assoluto. Qualsiasi ASIN con un ROI stimato **inferiore all'8%** viene scartato a prescindere dal punteggio VGP. Protegge il capitale da oscillazioni di prezzo Amazon o fee impreviste, garantendo un margine di sicurezza minimo.

Soglia default `0.08`. Configurabile dal cruscotto (L10 chiusa Round 5 — *"soglia 8% configurabile dal cruscotto, persistita in DB come config"*); la persistenza è già pronta in `config_overrides` (CHG-012). Questo CHG **non** integra ancora il config layer: la soglia entra come parametro funzione con default `0.08`. Il config layer (pydantic-settings → injection automatica) è scope futuro.

## Why

ROI scalare è disponibile da CHG-026 ma è solo un valore — non filtra ancora niente. Il Veto R-08 è il **primo filtro applicativo** in fila al Tetris allocator: protegge il capitale prima che il VGP score venga calcolato. Senza questo gate, ASIN con ROI -10% potrebbero comunque entrare nel ranking VGP (se gli altri due termini sono alti), violando R-08.

Inaugurare `vgp/` con il veto (anziché con normalize/score) è scelta deliberata di scope minimo: il veto è 1 funzione, 1 file, dipendenze zero. Inaugurare con normalize avrebbe richiesto Numpy/pandas + DataFrame di sessione, una fase a più variabili. Il veto si guadagna il primo posto perché è il pezzo più piccolo e indipendente del cluster `vgp/` — pattern coerente con CHG-022 (formulas inaugurato con la formula più atomica, fee_fba).

### Decisioni di design

1. **Predicato `is_vetoed_by_roi`**: nome positivo rispetto al concetto R-08 ("VETO ROI" → la funzione restituisce True quando il veto si applica). Convenzione idiomatica: `is_<condizione>` ritorna True quando la condizione è verificata. Alternativa `passes_roi_veto` scartata: meno coerente con il nome "VETO".
2. **Soglia parametro con default 0.08**: il default è ancorato a R-08 verbatim. La configurabilità (L10) entra al call site: il futuro config layer leggerà `config_overrides` e passerà la soglia esplicitamente. Approccio "explicit injection" preferito su global state.
3. **Validazione threshold in `(0, 1]`**: zero proibito (vorrebbe dire "nessun veto" — usare un altro filtro o disattivare il veto al call site, non con threshold=0). Negativo proibito (matematicamente assurdo: sarebbe sempre False). Maggiore di 1 proibito (matematicamente legittimo ma significherebbe "ROI > 100%" → veto sempre, indistinguibile dal disattivare la pipeline). Threshold==1.0 ammesso (boundary) ma di scarso uso pratico.
4. **`roi_value` non validato**: può essere qualsiasi float, inclusi negativi. La sua semantica è già stata gated da `roi()` di CHG-026.
5. **Versione vettoriale fuori scope**: il listino di sessione (DataFrame con N ASIN) si filtra con la versione vettoriale `vgp/normalize.py` o `vgp/score.py` futuro. Il predicato scalare di questo CHG è utile per:
   - test unit di logica isolata,
   - debug singolo ASIN,
   - eventuale veto-by-veto in pipeline non vettoriali (es. lookup loop).
6. **Telemetria fuori scope**: l'evento canonico `veto_roi_applied` (con campi `asin`, `roi_value`, `threshold`, `vetoed`) andrà inscritto nel catalogo `observability/` (CHG-006) con il primo uso reale del predicato in pipeline. Anticiparlo qui sarebbe codice non collegato.
7. **Costante `DEFAULT_ROI_VETO_THRESHOLD`**: esportata in `__all__` di `vgp/__init__.py` perché altri call site ne avranno bisogno (es. test di altre logiche, UI cruscotto come default).

## How

### `src/talos/vgp/veto.py`

```python
DEFAULT_ROI_VETO_THRESHOLD: float = 0.08

def is_vetoed_by_roi(roi_value: float, threshold: float = DEFAULT_ROI_VETO_THRESHOLD) -> bool:
    if not 0 < threshold <= 1: raise ValueError(...)
    return roi_value < threshold
```

R-08 verbatim ("inferiore all'8%") → strict `<`. ROI esattamente 0.08 **passa** (boundary inclusivo).

### Test plan (11 test)

1. `test_default_threshold_is_eight_percent` → `DEFAULT_ROI_VETO_THRESHOLD == 0.08` (ancora R-08 verbatim)
2. `test_vetoes_below_default_threshold` → `roi=0.07` → True
3. `test_passes_at_default_threshold` → `roi=0.08` → False (boundary R-08 verbatim)
4. `test_passes_above_default_threshold` → `roi=0.15` → False
5. `test_vetoes_negative_roi` → `roi=-0.10` → True (loss propagato)
6. `test_custom_threshold_below_default` → `roi=0.05, threshold=0.04` → False
7. `test_custom_threshold_above_default_vetoes` → `roi=0.15, threshold=0.20` → True
8. `test_threshold_one_inclusive` → `threshold=1.0, roi=1.0` → False (boundary superiore inclusivo)
9. `test_raises_on_zero_threshold` → `threshold=0` → ValueError
10. `test_raises_on_negative_threshold` → `threshold=-0.05` → ValueError
11. `test_raises_on_threshold_above_one` → `threshold=1.5` → ValueError

### Out-of-scope

- **Telemetria evento canonico** `veto_roi_applied`: scope futuro (definizione in observability + uso reale in pipeline).
- **Soglia da config layer** (pydantic-settings + lookup `config_overrides`): scope futuro CHG.
- **Versione vettoriale** (filtro su DataFrame di sessione): scope futuro `vgp/score.py` o equivalente.
- **Veto multiplo** (più filtri di sbarramento R-XX): scope futuro se emergeranno altri veti.
- **Test integration con DB** (lettura soglia da `config_overrides` runtime): scope futuro.

**Rischi residui:**

- Threshold come parametro con default 0.08 è "soft": un call site malizioso può passare 0.0 (… che però viene gated da ValueError). La protezione del capitale dipende dalla disciplina del call site.
- Boundary `roi == threshold` è "passa" (R-08 verbatim "inferiore all'8%" = strict `<`). Se in futuro il Leader vuole "≤ 8% scarta" è errata di ADR-0018 + cambio del segno di confronto.
- Nessuna telemetria significa che il veto applicato non lascia traccia. Mitigazione: scope futuro inscritto sopra.
